Concepts encountered while building M1, in plain language.

## Python concepts

### "kwargs" (keyword arguments)

There are two ways to pass values into a function:

```python
greet("Praveen", "hello")                # positional: matched by ORDER
greet(name="Praveen", message="hello")   # keyword: matched by NAME
```

A `name=value` pair at the call site is a **keyword argument** — "kwarg"
for short. Nothing more mysterious than "an argument passed by name".

The `**` operator unpacks a dict into keyword arguments:

```python
data = {"name": "Praveen", "message": "hello"}
greet(**data)
# Python rewrites this as: greet(name="Praveen", message="hello")
```

Step by step: `**data` takes each key/value pair in the dict and turns it
into `key=value` at the call site. The dict keys MUST match the function's
parameter names, or you get a TypeError.

Why our tests use this: you cannot edit one argument of a normal call,
but you CAN edit a dict before unpacking it.

```python
kwargs = make_valid_kwargs()    # full set of valid fields, as a dict
kwargs["confidence"] = 1.5      # break exactly ONE field
FailureAnalysis(**kwargs)       # should now fail validation
```

Start valid, break one thing, assert it's rejected. When the test fails,
you know exactly which rule was violated.

### `Literal` — "only these exact values"

```python
from typing import Literal

Category = Literal["flaky", "product bug", "unknown"]
```

A `Literal` type means: this value must be EXACTLY one of the listed
strings. Not "some string" — one of these specific strings.

In a Pydantic model, `suspected_category: Category` makes Pydantic reject
any other value at construction time:

```python
FailureAnalysis(suspected_category="network issue", ...)  # ValidationError
```

Why we care: LLMs improvise. If the model invents a category name that's
almost-but-not-quite right, every downstream filter ("show all network
failures") silently misses it. Literal turns improvisation into a loud
error at the boundary instead of silent data corruption.

### Pydantic `Field` — two different jobs

`Field()` attaches extra rules or behavior to a model field. We used it
two ways:

**1. Constraints (validation rules):**

```python
confidence: float = Field(ge=0.0, le=1.0)
```

`ge` = greater-or-equal, `le` = less-or-equal. So confidence must be
between 0 and 1 inclusive. `1.5` or `-0.1` → ValidationError.
The rule lives ON the model, so it's enforced everywhere the model is
built — CLI, API, tests, anywhere.

**2. Safe defaults for lists (`default_factory`):**

```python
evidence: list[str] = Field(default_factory=list)
```

This means: "if the caller doesn't provide evidence, call `list()` to
make a FRESH empty list for this instance."

Why not just write `evidence: list[str] = []`? Because in plain Python,
a default value is created ONCE and shared by every call that omits it.
Two objects would share the SAME list — append to one, it appears in the
other. A classic bug. `default_factory=list` runs `list()` fresh each
time, so every instance gets its own list. (Pydantic actually guards
against the shared-default trap internally, but `default_factory` is the
explicit, always-correct habit.)

### `del` — remove a key from a dict

```python
del kwargs["evidence"]
```

Removes the key entirely — as if it was never passed. Different from
setting it to `None` (key present, value None) or `[]` (key present,
empty list). Our defaults test uses `del` because we want to test what
happens when the field is genuinely absent.

### Negative tests with `pytest.raises`

```python
with pytest.raises(ValidationError):
    FailureAnalysis(**bad_kwargs)
```

Asserts the code inside the block THROWS. If no exception is raised, the
test FAILS. Same instinct as API testing: don't just test the 200s, test
that garbage input gets a 400.

**Trap discovered while doing TDD:** a test expecting ValidationError can
pass for the WRONG reason. Our reject-tests passed even before the new
model existed — because the OLD model rejected the data too (missing
`explanation` field), not because the category/confidence rules worked.
pytest can't tell the difference between the right error and any error of
the same type. Lesson: when the red phase looks "too green", read WHY the
test passed before trusting it.

## AI/design concepts

### Text blob vs structured output

"Blob" = informal term for one unstructured lump of data. M0's
`explanation` was a text blob: category, evidence, and advice all mushed
into one string. Humans can read it; programs can't query it.

M1's structured shape exists to answer questions a blob can't:

| Question | Needs |
|---|---|
| "Show only product bugs" | `suspected_category` (filterable) |
| "How sure is the model?" | `confidence` (sortable number) |
| "What do I check first?" | `next_steps` (explicit list) |
| "12 flaky, 3 env, 1 real bug this week" | machine-readable categories |

Free text is for humans. Structured data is for programs.

### Validation at the boundary

The model (`FailureAnalysis`) is the CONTRACT for what the LLM must
return. Pydantic enforces it at construction time — bad category, out of
range confidence, wrong types all fail loudly at the edge of the system,
before corrupt data can spread inward. Later M1 tasks build the machinery
to (a) tell the LLM about this contract in the prompt and (b) retry when
it violates it.

### Breaking changes as a feature

Adding required fields to FailureAnalysis broke core.py, the CLI, and the
API tests immediately. That's the type system working FOR us: every
consumer of the old shape is forced to surface, so nothing silently keeps
using the stale format. The failures are the to-do list for the rest of
the milestone.

## Task 2 — schema-in-prompt

### `get_args` — reading values off a Literal at runtime

```python
from typing import get_args

Category = Literal["flaky", "product bug", "unknown"]
get_args(Category)   # -> ("flaky", "product bug", "unknown")
```

A `Literal` type is normally just for the validator. `get_args` lets
running code read the allowed values off it as a plain tuple.

Two places use it:

- `core.py` builds the category list INTO the prompt from the model's
  own Literal — the prompt and the validator share one source of truth
  and can never drift apart.
- The test loops `for category in get_args(Category)` and asserts each
  appears in the prompt — add a 9th category to the model and the test
  automatically starts checking for it. No duplicated list in the test.

Why it matters: if the same list lives in two places (model + prompt, or
model + test), someone eventually updates one and forgets the other.
Deriving one from the other makes that bug impossible.

### `in` — substring check on strings

```python
"Timeout" in prompt        # True if the text appears anywhere in prompt
```

For strings, `in` asks "does this text appear anywhere inside that
string?" Returns True/False. Combined with assert:
`assert "Timeout" in prompt` = "fail the test unless the prompt
mentions Timeout".

### f-strings building quoted text

```python
field = "summary"
f'"{field}"'    # produces:  "summary"   (WITH the double quotes)
```

An f-string substitutes `{field}` with its value; the surrounding `"..."`
characters are just literal text in the template. The test uses this to
assert the prompt contains `"summary"` quoted like a JSON key — stricter
than the bare word, which could appear in an ordinary sentence.

### Module-private names: the leading underscore

```python
_CATEGORY_LIST = ", ".join(...)
```

A leading `_` on a name is Python convention for "internal to this file —
don't import or rely on it from outside". Not enforced by the language,
but respected by tools and readers.

### `", ".join(...)` — glue a list into one string

```python
", ".join(f'"{c}"' for c in get_args(Category))
# -> "product bug", "api outage", ..., "unknown"
```

`join` takes many strings and glues them into one, with the separator
(here `", "`) between each pair. The part inside is a generator
expression: "for each category c, produce it wrapped in quotes".

### Test the contract, not the wording

The prompt tests check WHAT the prompt contains (facts, category names,
field names, the word JSON) — never exact sentences. So the prompt's
phrasing can be freely improved without breaking tests; they only fail
if a fact, category, or field genuinely goes missing. Tests that pin
exact wording break on every harmless edit and teach people to ignore
failures.

### Prompt-engineering takeaways (first real prompting of the project)

- **With a plain-text LLM API, structure only comes from asking for it
  precisely**: name every key, list every allowed value, state the range,
  and say "ONLY a JSON object, no markdown fences, no extra text".
- **Never ask the LLM for facts you already have.** We know `test_title`
  from the report; asking the model to echo it back invites mangling.
  Code fills it in after parsing.
- **Instructions reduce misbehavior; they don't eliminate it.** We tell
  the model "no markdown fences", but Task 3 still strips fences
  defensively. Belt and suspenders.

## Task 3 — defensive JSON parsing

### "strip" — removing unwanted stuff from the edges

Plain English: strip = remove wrapping, keep the thing inside (like
stripping paint or a wrapper). Python's built-in:

```python
"  hello \n".strip()   # -> "hello"
```

`.strip()` removes whitespace (spaces/tabs/newlines) from the START and
END of a string only — never the middle.

Our `_strip_markdown_fences` extends the idea to a bigger wrapper: LLMs
often return

    ```json
    { ... }
    ```

The fence lines are chat-UI decoration, not JSON — `json.loads` chokes
on them. The helper unwraps: strip whitespace → if it starts with
` ``` ` drop the first line → `removesuffix("```")` chops the closing
fence → strip again. The JSON in between is untouched.

### String slicing — `text[n:]`

```python
text = "abcdef"
text[2:]    # -> "cdef"   (from position 2 to the end)
```

`text[first_newline + 1:]` = "everything after the first newline" —
i.e. drop the opening ```json line, keep the rest. Positions count
from 0.

### `json.loads` — text that LOOKS like JSON -> real Python data

```python
import json
json.loads('{"a": 1}')    # -> {"a": 1}  (a real dict)
```

"loads" = load from string. The LLM gives us TEXT; `json.loads` turns
it into a dict/list/numbers we can actually use. Invalid JSON raises
`json.JSONDecodeError`. (Counterpart: `json.load`, no "s", reads from
a file — the report parser uses that one.)

### `try/except` — attempt, and handle the blow-up

```python
try:
    data = json.loads(cleaned)          # attempt this
except json.JSONDecodeError as exc:     # if THAT error happened...
    raise ValueError(f"not JSON: {exc}") from exc   # re-throw as ours
```

Why re-throw? `parse_analysis` can fail two ways — not JSON at all
(json library's error) or JSON that breaks the schema (Pydantic's
error). Callers shouldn't care about either library: they catch ONE
type, `ValueError`, with a readable message. `from exc` chains the
original error so tracebacks still show the root cause.

### `:.0%` format spec

```python
f"{0.5:.0%}"    # -> "50%"
```

Inside an f-string, `:` starts formatting instructions; `.0%` means
"as a percentage with 0 decimal places". Used by the CLI to print
confidence.

### The big idea: never trust, always verify (LLM edition)

**Q: how are we sure the LLM sticks to the structure we asked for?**
**A: we are NOT sure — ever.** That's the mental shift from normal APIs.
A REST API with a schema returns that schema every time, guaranteed by
code. An LLM generates text word-by-word, probabilistically. A good
model follows the prompt ~95-99% of the time, but can still: wrap JSON
in fences, add "Sure! Here's the analysis:" before it, invent a
category, return confidence 1.2, or refuse outright.

So the design is layered defense, same instinct as validating input on
a public API:

| Layer | Role | API-testing analogy |
|---|---|---|
| Prompt (Task 2) | ASK for the right shape | API docs for clients |
| Parse + validate (Task 3) | VERIFY every reply at the boundary | server-side 400 on bad input |
| Retry loop (Task 4) | on failure, re-ask with the error message | client retry with corrected request |

### No new LLM magic — just engineering around one dumb call

Task 3 contains ZERO new LLM methods. The only LLM call in the codebase
is still `gateway.generate(prompt)` — text in, text out. Everything new
is plain standard-library Python applied to the text that comes back.
Much of "AI engineering" is exactly this: ordinary software engineering
wrapped around one probabilistic text call.

### The three tasks in one breath

- Task 1: DEFINED the structure (FailureAnalysis = the contract)
- Task 2: TOLD the LLM about it (schema in the prompt)
- Task 3: CHECKED the reply against it (unwrap -> parse -> validate ->
  guaranteed-valid object or loud ValueError)

One-liner: **we built the checkpoint where untrusted LLM text becomes
trusted, validated data — or gets rejected.** Downstream code never
worries about the LLM misbehaving again.

### Ripple effects worth remembering

- **FakeGateway's canned reply had to become valid JSON** — a fake must
  imitate a WELL-BEHAVED real dependency, or every test downstream of
  it breaks for the wrong reason.
- **api.py needed zero changes** — it returns whatever FailureAnalysis
  is (`response_model=list[FailureAnalysis]`), so the new shape flowed
  through automatically. Declare the contract once, consumers follow.

## Task 4 — the retry loop (self-correction)

### The insight that makes LLMs different from flaky services

A 503 from a REST API won't change because you describe the error back
to the server. An LLM WILL: tell it "your reply was invalid: confidence
must be <= 1.0. Try again." and it genuinely produces corrected JSON.
The **self-correction loop** is the standard industry pattern for
structured outputs:

1. send prompt, try to parse
2. success -> return (happy path costs exactly ONE llm call)
3. failure -> re-prompt with the exact validation error + the original
   task + "ONLY the corrected JSON"
4. after max_attempts, give up LOUDLY (raise) — never loop forever,
   never fail silently

### Scripted test double

To test a retry loop, the fake must misbehave FIRST and comply AFTER —
one canned reply can't do that. FakeGateway now takes a script:

```python
FakeGateway(responses=["garbage", VALID_JSON_REPLY])
# call 1 -> "garbage", call 2 -> valid JSON, further calls -> repeat last
```

Same instinct as testing flaky-dependency handling in any framework:
the double plays a sequence, the test asserts recovery. Design detail:
when the script is exhausted, the LAST reply repeats — an extra call
never crashes a test for a fake-internal reason.

The four retry tests, in words:
- first reply good -> exactly 1 gateway call (retries are free when
  unneeded)
- bad then good -> recovers, 2 calls
- the SECOND prompt sent (`gateway.calls[1]`) mentions "invalid",
  "JSON", and repeats the original task — an LLM can't fix what it
  can't see
- all garbage -> ValueError after exactly max_attempts calls

### Python concepts met here

**Conditional expression (one-line if/else):**

```python
self.responses = responses if responses is not None else [response]
```

Reads: "use `responses` if it was given, otherwise `[response]`".
Internally the fake has ONE mode — always a list; a single reply is
just a script of length 1.

**`min()` to cap an index:**

```python
index = min(len(self.calls) - 1, len(self.responses) - 1)
```

Call 1 -> index 0, call 2 -> index 1, ... but never past the end of
the script. The `min` IS the "repeat last when exhausted" rule — no
if-statement needed.

**`for _ in range(n)` — loop n times, ignore the counter:**

`range(3)` yields 0, 1, 2. Naming the variable `_` signals "I never
use this value — I only care that the body runs at most n times."

**`return` inside a loop** exits the whole function immediately —
that's how the happy path costs one call: parse succeeds, return fires,
no retry code runs.

**The exception is data.** `except ValueError as exc:` binds the error
object to a name; `f"...invalid: {exc}"` embeds its message into the
repair prompt. The error isn't just a signal to stop — its CONTENT
becomes the LLM's feedback.

**Import-error interrupts collection.** When `generate_analysis` didn't
exist yet, pytest couldn't even import test_core.py — so NO tests ran,
not even unrelated ones. One bad import silences a whole module; the
"3 failed" you expected shows up as "1 error during collection".

### Subtle bug avoided: repair prompts must not nest

The repair prompt is built from `prompt` (the ORIGINAL), not
`current_prompt` (the previous repair). Otherwise attempt 3's prompt
would wrap attempt 2's repair wrapping attempt 1's task — Russian
dolls of "your reply was invalid", growing every round and burying
the actual task.
