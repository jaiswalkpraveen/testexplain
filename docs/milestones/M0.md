## Python concepts learned

### Packages, modules, imports
```python
from testlens.models import FailureContext
#    └ package ┘ └module┘        └ class ┘
```
- **package** = a folder (`src/testexplain/`).
- **module** = a file (`models.py`).
- Dotted path = folder/file address, dots separate levels.
- This is just *where the thing lives* — NOT inheritance.

| Python | Java | Node / TS |
|---|---|---|
| `from testlens.models import FailureContext` | `import com.testlens.models.FailureContext;` | `import { FailureContext } from "./testlens/models";` |

### Inheritance (the real parent/child)
```python
class FailureContext(BaseModel):
#     └ child ┘     └ parent ┘
```
- Parentheses after a class name mean **"extends"**.
- `FailureContext` **is a** `BaseModel` and inherits all its machinery.
- Parent = `BaseModel` (from pydantic); child = `FailureContext` (mine).

| Python | Java | Node / TS |
|---|---|---|
| `class FailureContext(BaseModel):` | `class FailureContext extends BaseModel {` | `class FailureContext extends BaseModel {` |

> Watch out: `( )` means **extends** when defining a class, but means **construct an object** when calling `FailureContext(test_title="x")`. Same symbol, two jobs.

### Required vs optional fields
```python
class FailureContext(BaseModel):
    test_title: str        # required (no default)
    duration_ms: int = 0   # optional (has a default)
```
- `field: type` with no `=` → **required**.
- `= value` → **optional**, uses the default when omitted.

| Concept | Python / Pydantic | Java | Node / TS (Zod) |
|---|---|---|---|
| Required field | field, no default | non-null field | `z.string()` |
| Optional w/ default | `duration_ms: int = 0` | `int duration = 0;` | `z.number().default(0)` |

## Pydantic deep-dive

### What it is
A library to declare the **shape** of data and **validate** it at runtime. `BaseModel` is the base class you inherit to get that behavior. Mental model for an SDET: **a Pydantic model is a built-in assertion on a data structure** — wrong shape fails loudly at the boundary.

Closest analog in other stacks: **Zod in TypeScript** (declare a schema, construction/parse throws a descriptive error listing all violations). Java needs an extra framework (Bean Validation / Hibernate Validator) to get the same.

### Validation is automatic on construction
- Creating the object — `FailureContext(test_title="x")` — runs validation inside `BaseModel.__init__`.
- I never call a validate method; just constructing triggers it.
- If required fields are missing → raises `ValidationError` listing **every** missing/invalid field at once.

Example error from omitting 3 required fields (`file`, `status`, `error_message`):
```
pydantic_core._pydantic_core.ValidationError: 3 validation errors for FailureContext
file
  Field required [type=missing, input_value={'test_title': 'x'}, input_type=dict]
status
  Field required [type=missing, input_value={'test_title': 'x'}, input_type=dict]
error_message
  Field required [type=missing, input_value={'test_title': 'x'}, input_type=dict]
```
- It does NOT complain about `error_stack` / `duration_ms` because those have defaults (optional).
- `input_value=...` shows what I actually passed; `input_type=dict` is how it received the keyword args internally.
- In the REPL, the error aborts the line — so running 3 broken lines shows the same error 3 times, each independent.

### `model_dump()` — separate and manual
- `model_dump()` does the OPPOSITE direction: model → plain `dict`. It is **opt-in** (only runs if I call `fc.model_dump()`).
- It was NOT involved in the validation error.

| Method | Direction | When | Purpose |
|---|---|---|---|
| `FailureContext(...)` | data in → model | automatic, on create | validate; raise `ValidationError` if bad |
| `fc.model_dump()` | model → dict out | only when called | serialize back to a plain dict |

| | Python / Pydantic | Java | Node / TS |
|---|---|---|---|
| Validate on create | automatic in `__init__` | constructor / validator | `schema.parse(data)` |
| Serialize out | `fc.model_dump()` | Jackson / `toMap()` | `JSON.stringify` / `.toObject()` |

### How to inspect `BaseModel`'s own code
All read-only and safe:
1. Find it on disk: `uv run python -c "import pydantic; print(pydantic.__file__)"` → `BaseModel` lives in `pydantic/main.py` inside `.venv/.../site-packages/pydantic/`.
2. Explore live: `help(BaseModel)` and `dir(BaseModel)` (list methods: `model_dump`, `model_validate`, `model_fields`, ...). `import inspect; inspect.getsourcefile(BaseModel)`.
3. Open the file from the path in step 1.
4. Browse on GitHub: `github.com/pydantic/pydantic` → `pydantic/main.py`.

> Note: Pydantic v2's fast validation engine is compiled **Rust** (`pydantic-core`), so the Python file shows the interface/orchestration, not the core engine. `dir()` = the menu of what a class can do (like IDE autocomplete); `help()` = the docs for each.

## Reading & parsing the JSON file: `json.loads(Path(path).read_text())`

One line, three operations. Read it **inside-out** (right to left):

```
Path(path)        → a Path object (a file LOCATION, not the contents)
  .read_text()    → reads the whole file → returns a str (raw text)
json.loads( ... ) → parses that str → returns a Python dict / list
data = ...        → creates the variable `data` and stores the result
```

### Where is `data` defined?
On this exact line — the `=` creates it. Python has no declaration keyword (no `var`/`let`/`const`, no `Map data =`). A variable exists the moment you assign to it.

| Language | Equivalent |
|---|---|
| Java | `Map<String,Object> data = new Gson().fromJson(text, Map.class);` |
| TS/Node | `const data = JSON.parse(text);` |
| Python | `data = json.loads(text)` |

### What type is `data`?
Whatever the JSON's **top level** is. `json.loads` maps JSON → Python:

| JSON top-level | Python type |
|---|---|
| `{ ... }` object | `dict` |
| `[ ... ]` array | `list` |
| `"text"` | `str` |
| `123` / `1.5` | `int` / `float` |
| `true`/`false` | `bool` |
| `null` | `None` |

`sample_report.json` starts with `{ "suites": [...] }` → an object → **`data` is a `dict`**. That's why the next line can call `data.get("suites", [])`. The type is inferred at runtime; it is NOT annotated in the code.

### Value-by-value trace
Given `path = "tests/fixtures/sample_report.json"` and file contents `{"suites": [{"title": "auth.spec.ts"}]}`:

```python
Path(path)            # PosixPath('tests/fixtures/sample_report.json')  ← location only, disk untouched
Path(path).read_text()# '{"suites": [{"title": "auth.spec.ts"}]}'      ← a str
json.loads(<that str>)# {'suites': [{'title': 'auth.spec.ts'}]}        ← a dict
data["suites"]        # [{'title': 'auth.spec.ts'}]                     ← key access now works
```

After step 2 it's still a **string** — `data["suites"]` would fail. Only after `json.loads` is it a real dict.

### `loads` vs `load` (the trailing `s` = string)
| Function | Input → Output | Use |
|---|---|---|
| `json.loads(s)` | **str** → dict/list | parse text you already have (our case) |
| `json.load(f)` | **file object** → dict/list | parse straight from an open file |

`Path(...).read_text()` already gave us a string, so we use `loads`. (`read_text` also opens AND closes the file for us — cleaner than `open()`.)

Cross-language: `json.loads` = `JSON.parse` (JS) = `new Gson().fromJson(...)` (Java). `read_text()` = `fs.readFileSync(path,"utf-8")` (Node) = `Files.readString(...)` (Java).

### Prove it in the REPL
```bash
uv run python -c "
from pathlib import Path
import json
p = Path('tests/fixtures/sample_report.json')
print(type(p))            # <class 'pathlib.PosixPath'>
text = p.read_text()
print(type(text))         # <class 'str'>
data = json.loads(text)
print(type(data))         # <class 'dict'>
print(list(data.keys()))  # ['suites']
"
```

## Task 4 — The LLM gateway seam (OOP + Protocol)

The gateway is the seam that lets tests use a fake LLM and production use real Claude, **without the core code knowing the difference**. Three pieces live in `src/testexplain/gateway.py`: the `Gateway` Protocol (contract), `FakeGateway` (test double), `AnthropicGateway` (real).

### `self` — the object a method was called on
`gw.generate("hi")` is secretly `FakeGateway.generate(gw, "hi")` — Python passes the object as the first arg, named `self` by convention. `self.x` = "this object's x".

| Python | Java | JS/TS |
|---|---|---|
| `self` (explicit param) | `this` (hidden) | `this` (hidden) |
| `self.response` | `this.response` | `this.response` |

### `__init__` — the constructor
Runs automatically once when you build an object: `FakeGateway(response="boom")`. Python makes a blank object, then calls `__init__(self, ...)` on it. `self.response = response` stores the arg as a permanent **attribute**. `__init__` is a *dunder* (double-underscore) method Python calls for you — never call it directly.

| Python | Java | JS |
|---|---|---|
| `def __init__(self, ...)` | `ClassName(...)` | `constructor(...)` |

### Default arguments
`def __init__(self, response="FAKE: ...")` → `response` is optional.
- `FakeGateway()` → uses the default (when a test doesn't care about the text, e.g. testing the `calls` spy).
- `FakeGateway(response="boom")` → override (when a test asserts on the exact returned text).

Python default arg = JS `constructor(response="...")` = **two overloaded constructors** in Java.

### Attribute vs local variable
- **Local variable** — lives only during one method call, then gone.
- **Attribute** (`self.calls`) — attached to the object, survives between calls.

`self.calls: list[str] = []` in `__init__` creates the field on the fly (no pre-declaration like Java's `private List<String> calls`). `self.calls` (inside) and `gw.calls` (outside) are **the same list on the same object**, seen from two places. That's why a test can read `gw.calls` after calling `gw.generate(...)`.

`self.calls` is a **spy/recorder**: `generate()` does `self.calls.append(prompt)`, so after two calls `gw.calls == ["A", "B"]`. Lets tests assert what was sent — with no network.

### Protocol — the contract, via duck typing
```python
from typing import Protocol

class Gateway(Protocol):
    def generate(self, prompt: str) -> str: ...
```
- `Protocol` = a **shape**, not a class you instantiate. Never write `Gateway()`.
- `...` (Ellipsis) = "intentionally empty body" (just a signature).
- **Duck typing / structural typing:** `FakeGateway` and `AnthropicGateway` satisfy `Gateway` simply by *having* a matching `generate` method. **No `implements`, no inheritance.** If it walks like a duck…

| | Python (structural) | Java (nominal) | TS |
|---|---|---|---|
| Declare | `class Gateway(Protocol):` | `interface Gateway {}` | `interface Gateway {}` |
| Opt in | nothing — match the shape | `implements Gateway` | nothing — match the shape |
| Enforced by | mypy (optional), not runtime | compiler, always | compiler (erased at runtime) |

**At runtime Python enforces nothing** — the Protocol exists for the type checker (mypy) and for humans. Its real job: document that the core analysis code depends on `.generate()` and nothing else.

### Lazy import (the key trick)
`AnthropicGateway.__init__` imports `anthropic` *inside* the method, not at the top of the file. So importing `gateway.py` for `FakeGateway` tests never needs the `anthropic` package or an API key. This is the same idea as card-eval's `get_chat_model()` lazily importing `langchain_openai`.

### card-eval ↔ TestLens
card-eval hides the swap behind `is_mock()` (an `if mock:` branch inside `llm.py`). TestLens expresses the same swap as a clean **Protocol with two implementations** — no branch, just hand the core code whichever gateway you want.

## Task 5 — Core analysis (build_prompt + analyze_report)

`src/testexplain/core.py` is the **reasoning layer**: parse → prompt → generate → wrap.

### f-strings and triple-quoted strings
`build_prompt` returns an `f"""..."""` string.
- The `f` prefix = **formatted string**: anything in `{...}` is evaluated and inserted. `{failure.test_title}` drops the value in.
- Triple quotes `"""..."""` = **multiline** literal (newlines kept as-is).

| Concept | Python | Java | JS/TS |
|---|---|---|---|
| Insert a value | `f"{x}"` | `String.format("%s", x)` | `` `${x}` `` |
| Multiline literal | `"""..."""` | text block `"""..."""` | `` `...` `` (backticks) |

Prompt-building is pure string work — **no LLM call** — so it's trivially testable (assert the error message is in the prompt).

### The seam payoff
`analyze_report(path, gateway: Gateway)` only ever calls `gateway.generate(prompt)`. It never mentions `FakeGateway`/`AnthropicGateway`. Tests pass a `FakeGateway`, so the whole pipeline runs with **no network, no API key**, yet every line is exercised. Same idea as card-eval's `is_mock()`, but expressed as a clean parameter instead of an `if mock:` branch.

### Why the test showed no output (important mental model)
A **test asserts, it doesn't print**. `analyze_report` ran, produced a `FailureAnalysis`, the test checked its values, then discarded it. Green = it worked; there's just no visible output by design.

Also: with `FakeGateway`, the "response" is whatever canned string you built it with — **not** a real LLM answer. It always returns that fixed text. Real explanations come from `AnthropicGateway` (Task 6 CLI / Task 7 API).

### How to see the result yourself
```bash
# 1. Quick run with the fake (no key)
uv run python -c "
from testlens.core import analyze_report
from testlens.gateway import FakeGateway
for r in analyze_report('tests/fixtures/sample_report.json', FakeGateway(response='hello from fake')):
    print(r.test_title, '->', r.explanation)
"

# 2. See every field of the result object (model_dump = Pydantic object -> dict)
uv run python -c "
from testlens.core import analyze_report
from testlens.gateway import FakeGateway
print(analyze_report('tests/fixtures/sample_report.json', FakeGateway())[0].model_dump())
"

# 3. See the actual prompt that gets sent to the LLM
uv run python -c "
from testlens.ingestion.playwright import parse_report
from testlens.core import build_prompt
print(build_prompt(parse_report('tests/fixtures/sample_report.json')[0]))
"
```
A **real** LLM answer arrives in Task 6: `testexplain analyze ./report.json` runs `analyze_report(..., AnthropicGateway())` and prints real explanations.

## Task 6 — CLI entry point (Typer)

`src/testexplain/cli.py` gives us `testexplain analyze ./report.json`. The `pyproject.toml` line
`testexplain = "testexplain.cli:app"` means "the `testexplain` command runs the `app` object in `cli.py`."

### Typer basics
Typer builds a CLI from a plain function: parameters become CLI args, and it handles `--help`, parsing, and errors.

| Concept | Typer (Python) | Node | Java |
|---|---|---|---|
| CLI framework | Typer | commander / yargs | picocli |
| A command | decorated function | `.command()` | `@Command` method |

### New Python syntax
- **Decorator `@app.command()`** — a function that wraps another to add behavior. `@app.command()` registers `analyze` as a CLI command. Python runs it as `analyze = app.command()(analyze)`. Unlike a Java `@Annotation` (passive metadata read via reflection), a Python decorator **actively runs** and can wrap/replace the function.
- **`fake: bool = False` → `--fake` flag** — Typer turns a bool param with default `False` into an optional on/off flag (`--fake` / `--no-fake`). A param with no default (`report_path: str`) becomes a required argument.
- **Ternary `X if cond else Y`** — Python's inline conditional. Java/JS write it `cond ? X : Y`.
- **`if __name__ == "__main__": app()`** — lets you run the file directly (`python cli.py`) as well as via the installed command. `__name__ == "__main__"` is true only when run directly, not when imported. (Java analog: `public static void main`.)

### The single-command gotcha (a real bug we hit)
With exactly ONE `@app.command()`, Typer **collapses** the command to the top level — so `testexplain analyze report.json` broke: `analyze` was parsed as the path and the real path became an "unexpected extra argument" (exit code 2).
**Fix:** add an `@app.callback()` (even an empty one). That gives the app a top-level command *group*, so `analyze` stays a real subcommand. `--help` then shows `Usage: testexplain [OPTIONS] COMMAND [ARGS]...` with `analyze` listed.

> Exit codes: `0` = success, `2` = CLI usage/parse error (bad args), and we made a missing API key exit `1` with a friendly message via `typer.Exit(code=1)` instead of dumping a traceback.

### Seeing real output (the M0 payoff)
```bash
uv run testexplain analyze tests/fixtures/sample_report.json --fake   # canned, no key
uv run testexplain analyze tests/fixtures/sample_report.json          # real Claude (needs ANTHROPIC_API_KEY)
```

### Testing a CLI
`typer.testing.CliRunner` invokes the command in-process and captures output:
```python
result = runner.invoke(app, ["analyze", str(FIXTURE), "--fake"])
assert result.exit_code == 0
assert "..." in result.stdout
```
Using `--fake` means the CLI test needs no API key and no network.

## Task 7 — HTTP API (FastAPI)

`src/testexplain/api.py` exposes the same pipeline over HTTP so a dashboard (or any client) can request explanations. M0 is one read-only endpoint:
```
GET /analyze?report_path=...&fake=true
```

### FastAPI basics
- `app = FastAPI(...)` creates the application (like `express()` in Node).
- `@app.get("/analyze")` — a decorator (same concept as Typer's) registering a function as an HTTP GET handler. `@app.post(...)` etc. for other verbs.
- **Function params become request inputs.** `def analyze(report_path: str, fake: bool = False)` → FastAPI reads `report_path` and `fake` from the query string automatically, with type conversion (`fake=true` → Python `True`). No default = required; with default = optional.
- **`response_model=list[FailureAnalysis]`** — FastAPI validates and serializes the return value against the Pydantic model, so you return Python objects and the client gets clean JSON. Same Pydantic models we defined in Task 2 double as the API schema.

| Concept | FastAPI (Python) | Node (Express) | Java (Spring) |
|---|---|---|---|
| App object | `FastAPI()` | `express()` | `@SpringBootApplication` |
| Route | `@app.get("/x")` | `app.get("/x", ...)` | `@GetMapping("/x")` |
| Query param | function arg | `req.query.x` | `@RequestParam` |
| Response schema | `response_model=` | manual `res.json()` | return type + Jackson |

### Error handling
`raise HTTPException(status_code=404, detail=...)` returns a proper HTTP error instead of crashing (500). We return **404** for a missing report file and **500** if `ANTHROPIC_API_KEY` is missing on a real run. This is the API analog of the CLI's friendly-error handling.

### Testing an API
`fastapi.testclient.TestClient` calls the endpoints in-process (no running server, no network):
```python
client = TestClient(app)
response = client.get("/analyze", params={"report_path": "...", "fake": "true"})
assert response.status_code == 200
assert response.json()[0]["test_title"] == "..."
```
(Same idea as Playwright's `request` fixture for API testing — in-process here, so it's instant.)

### Running the server for real
```bash
uv run uvicorn testexplain.api:app --reload
# then visit http://127.0.0.1:8000/docs  (auto-generated Swagger UI)
```
`uvicorn` is the ASGI server that actually runs the FastAPI app; `--reload` restarts on code changes. `/docs` is FastAPI's free interactive API explorer.
